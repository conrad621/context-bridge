Local-first session bridge

跨编码 Agent 无缝迁移会话上下文。

Context Bridge 在 Claude Code、Codex 和 MCP 工具之间转换 JSONL 会话历史， 保留足够的操作轨迹，让你换工具时不用丢掉上下文。

开始使用 查看 npm
context-bridge translate
Claude Code JSONL ingest
Canonical Session normalize
Codex JSONL render 
$ ctxb translate <session-id>
created target session
resume: codex resume ...
本地优先 不依赖托管服务
双向转换 Claude Code 与 Codex
MCP 可调用 结构化 session 工具

Why it matters

会话不是聊天记录，而是工作现场。

编码 Agent 的 session 里包含用户意图、工具调用、shell 命令、文件操作、工作目录、 时间戳和标题。Context Bridge 把这些轨迹规范化成统一模型，再渲染到目标工具可以恢复的格式。

Pain points

多 Agent 工作流最痛的，不是启动新工具。

真正的损耗发生在切换之后：上下文散了、轨迹断了、重复解释开始了。 Context Bridge 处理的是这段隐形成本。

Lost context

上下文被锁在各自格式里

Claude Code 和 Codex 都记录 JSONL，但结构不同。换工具时，历史通常只能靠复制摘要。

Broken trace

工具调用和文件操作很难带走

一次真实编码会话包含 shell、patch、读写文件、工作目录和错误输出，纯聊天导出不够恢复现场。

Manual handoff

交接靠人工整理，越久越不可靠

长任务、跨天任务、多人协作时，手写 handoff 容易漏掉关键命令和决策路径。

Sync loops

自动同步容易生成重复链

如果不知道哪些 session 是生成物，批量同步会把迁移产物再次迁移，污染本地历史。

01

Translate

生成带来源标记的迁移 session，后续可清理、去重，并避免被再次链式同步。

02

Copy

一次性复制成新的目标 Agent session，不写入同步追踪元数据，表现更接近原生会话。

03

Sync / Watch

扫描最近原生 session，按方向批量生成确定性目标 session，适合长期双工具工作流。

Capabilities

为跨 Agent 工作流准备的基础设施。

Canonical IR

把不同 harness 的事件规范化成统一 Session 模型，减少格式耦合。

Adapter Boundary

Claude Code 和 Codex 各自维护 ingest/render 逻辑，转换路径清晰可扩展。

Provenance-aware Sync

生成 session 可识别、可跳过、可清理，避免同步循环和重复链路。

Session Index

跨 harness 列出、查看和筛选本地 session，快速找到要迁移的上下文。

Hook Integration

支持 Claude Code Stop hook 与 Codex notify hook，轻量触发自动同步。

MCP Server

通过 stdio 暴露 session 查询、转换和恢复命令准备能力。

Case tours

更具体的使用场景，一眼看懂它该放在哪里。

Case 01

长任务做到一半，换 Agent 继续收尾

你已经在 Claude Code 里完成需求理解、文件搜索、局部实现和几轮测试，但后半段想交给 Codex 做类型错误修复、测试补齐或最终验证。此时不需要重新解释项目背景，直接把当前 session 转过去。 

ctxb list --harness claude-code
ctxb smoke <session-id>
ctxb translate <session-id> --to codex
Case 02

复用一次高质量调研，但不建立同步关系

某次会话里已经沉淀了架构约束、关键文件、风险点和命令结果。你只想让另一个 Agent 从这里开始， 但不希望它被后续 clean/dedupe 当成生成物处理。使用 copy 创建独立的新会话。 

ctxb copy <session-id>

# no provenance markers
# fresh target session id
Case 03

每天在两个 CLI Agent 之间切换

白天用一个 Agent 快速执行，晚上用另一个 Agent 做审查、总结或继续实现。sync/watch 会扫描最近原生 session，并用 fingerprint 和来源标记避免重复生成与链式迁移。 

ctxb sync --direction both --days 14
ctxb watch --direction both --days 1 -i 30
Case 04

把 session 能力交给 MCP 自动化工具

MCP host、桌面自动化或本地编排工具不应该硬编码 Claude Code/Codex 的 session 路径。 通过 MCP server，它们可以列出会话、触发转换，并拿到可执行的恢复命令。 

ctxb mcp config-snippet
ctxb mcp serve
Case 05

发布前保留可追溯的审查链路

一个 Agent 完成实现后，另一个 Agent 负责 code review、风险复核和验收记录。translate 生成的目标 session 带来源元数据，后续能识别它来自哪次原始工作，而不是混进普通本地历史。 

ctxb inspect <source-session>
ctxb translate <source-session>
ctxb inspect <target-session>
Case 06

清理迁移产物，保持本地历史干净

试过几次迁移、同步或 watch 后，本地可能产生多份目标 session。因为生成物带有来源标记， clean 和 dedupe 可以先 dry-run，再删除可确认的迁移产物。 

ctxb clean --dry-run
ctxb dedupe --dry-run

# rerun without --dry-run when ready

Usage guide

第一次使用，从保守路径开始。

推荐先列出会话，再 smoke 预检转换结果。确认恢复命令和目标方向都正确后， 再执行 translate、copy 或 sync。

  1. 01

    安装 CLI

    全局安装后同时获得 context-bridge 和短别名 ctxb。

    npm install -g @mmmjk/context-bridge
ctxb --help
  2. 02

    找到要迁移的 session

    先用较短时间窗口浏览最近会话，必要时再扩大 --days。

    ctxb list --harness both --days 7 -n 20
ctxb inspect <session-id>
  3. 03

    先做 smoke 预检

    smoke 会验证转换并打印恢复命令，不执行真实模型，也不需要立即切换工具。

    ctxb smoke <session-id>
ctxb smoke <session-id> --to codex
  4. 04

    选择迁移、复制或同步

    需要可追踪迁移用 translate；需要干净独立起点用 copy；需要长期镜像用 sync/watch。

    ctxb translate <session-id>
ctxb copy <session-id>
ctxb sync --direction both --days 14
  5. 05

    执行输出的 resume 命令

    translate 会输出目标 Agent 的恢复命令。把提示词替换成下一步任务即可继续工作。

    # printed by translate
codex resume ... "<your next prompt>"

选择建议

想保留来源关系
用 translate。生成物会被标记，便于清理、去重和避免循环。
只想复制一次
用 copy。它不写追踪元数据，更适合作为新的原生起点。
想自动镜像最近会话
用 sync 或 watch，并从较短 --days 开始。
担心污染本地历史
先跑 clean --dry-run 和 dedupe --dry-run 查看会删什么。

CLI first

几个命令，把上下文带到下一个工具。

主命令是 context-bridge，同时提供更短的 ctxb 别名。 所有核心能力都围绕本地文件读写完成。 

ctxb list --harness both --days 30 -n 20
ctxb inspect <session-id>
ctxb smoke <session-id>
ctxb translate <session-id>
ctxb copy <session-id>

# fresh target session
# no sync tracking metadata
ctxb sync --direction both --days 365
ctxb watch --direction both --days 1 -i 30
